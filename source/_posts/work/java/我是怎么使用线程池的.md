## 简单的说明

首先给出声明的方法，也就是线程池的构造器。

![image-20230926151418320](%E6%88%91%E6%98%AF%E6%80%8E%E4%B9%88%E4%BD%BF%E7%94%A8%E7%BA%BF%E7%A8%8B%E6%B1%A0%E7%9A%84/image-20230926151418320.png)

- corePoolSize：线程池中一直存活的线程数量

- maximunPoolSize：线程池中最多存活的线程数量

- keepAliveTime：当存活线程数量大于核心线程数量时，多余线程的存活时间，空闲时间超过这个数值则销毁线程

- unit：时间单位

- workQueue：工作队列，必须是阻塞的

- threadFactory：线程工厂，用于新建线程时给线程添加一些配置。最常用的就是设置线程名。

- handler：当任务被拒绝时的后续处理。默认实现是任务被拒绝时会抛出一个异常。

线程池还有很多构造器，其他构造器大多是封装了一些默认实现，比如给了默认的threadFactory或者handler。

---

线程池有一个很反直觉的逻辑：只有任务队列满之后，才会创建新线程。

为什么这么设计？我们只需要记住：创建线程、销毁线程是一个性能开销很大的工作，我们应该尽量复用线程。

因此，**除非任务队列已满，否则都不会创建线程**。

下面来看一个实际使用时线程池的声明。

```
ThreadPoolExecutor EXECUTOR = new ThreadPoolExecutor(1, 1, 10, TimeUnit.SECONDS, new ArrayBlockingQueue(1), r -> new Thread(r, "cache-clean-thread"), new ThreadPoolExecutor.DiscardOldestPolicy());
```

我们来模拟一下这个线程池的处理逻辑，假设有耗时很长的任务：

- 第一个任务来到线程池，任务直接由核心线程执行。

- 第二个任务来到线程池，没有空闲线程，进入任务队列。

- 第三个任务来到线程池，没有空闲线程，任务队列也满了。这时候假如最大线程数大于核心线程数，则应该要创建一个新线程来执行任务，而当前任务则进入任务队列。但是我们不允许创建更多的线程，于是这个任务会被拒绝，进入handler的后置处理逻辑。我们直接使用了`new ThreadPoolExecutor.DiscardOldestPolicy()`作为后置处理逻辑，这个方法会移除最老的任务，然后把当前任务放到队列末尾。于是这个任务进入队列，最老的任务被移除，由于刚好我们队列为1，所以结果就是正在等待的任务被新来的任务替换了。

可以看到，线程池处理的关键是workQueue，threadFactory和handler这三个参数，通过设定这三个参数，我们可以把线程池玩出各种花样来。

接下来的章节，就主要针对线程池的使用了。

## 线程池的使用

### 命名规定

我们要求使用线程池时必须重写threadFactory来命名线程。在命名线程时说明线程大概要处理的任务，可以有效提高后续通过日志排查问题的效率。

以下是命名的示例

```
/**
 * 创建读线程池
 */
private static final ThreadPoolExecutor READ_EXECUTOR = 
	new ThreadPoolExecutor(4, 4, 30, TimeUnit.SECONDS, 
		new LinkedBlockingQueue(400), 
		new ThreadFactory() {
			private final AtomicInteger threadNumber = new AtomicInteger(1);

			@Override
			public Thread newThread(Runnable r) {
				return new Thread(r, "cache-read-thread-" + threadNumber.getAndIncrement());
			}
		});
```

### 提交任务

有两个方法：excuse和submit

区别在于submit之后可以获得一个Future，在后续逻辑中监控任务是否执行完，并获得其堆栈。

excuse方法执行完之后当前线程就无法感知这个任务是否执行完了，这在大多数情况下是不可接受的。

一般是建议使用submit。

案例：

```
Runnable writeTask1 = () -> {
// do something
};
Runnable writeTask2 = () -> {
// do something
};
int writeThreadCount = 4;
Future[] writeFutures = new Future[writeThreadCount];
// 提交任务到线程池
for (int i = 0; i < writeThreadCount; i++) {
	writeFutures[i] = WRITE_EXECUTOR.submit(i % 2 == 0 ? writeTask1 : writeTask2);
}        
// 等待所有线程执行完毕
for (Future future : writeFutures) {
	future.get();
}
```

### 拒绝策略

一般来说，拒绝策略可以通过设置workQueue和handler两种方式来修改。

ThreadPoolExecutor中已提供了四种内置策略，已经覆盖了绝大部分场景

```
// 在当前线程执行任务
new ThreadPoolExecutor.CallerRunsPolicy()
// 抛出任务被拒绝的异常
new ThreadPoolExecutor.AbortPolicy()
// 忽略最新提交的任务
new ThreadPoolExecutor.DiscardPolicy()
// 忽略最老提交的任务
new ThreadPoolExecutor.DiscardOldestPolicy()
```

以下列举两种特殊的策略

#### 不拒绝任何任务

不希望有任何任务被拒绝，需要设置workQueue为LinkedBlockingQueue，并且不设置长度限制。但要注意人物堆积时可能导致内存溢出。

同时，由于线程池的创建线程只发生在任务队列已满时，这种用链表作为workQueue的线程池将不会创建任何新的线程。

以下是示例：

```
private static final ThreadPoolExecutor notifyExecutor =
        new ThreadPoolExecutor(5, 5, 10, TimeUnit.SECONDS, new LinkedBlockingQueue<>(),
            new ThreadPoolExecutor.AbortPolicy());
```

由于没有任何任务被拒绝，handler其实也没有什么用了。

#### 任务溢出时让工作线程参与任务，且保证顺序

这是一种特殊情况下出现的需求，要求任务能保持一个大概的顺序（有时间戳），且数据不能丢失。

下面是代码示例：

```
private static final ThreadPoolExecutor NOTIFY_EXECUTOR =
    new ThreadPoolExecutor(1, 1, 10, TimeUnit.SECONDS, new ArrayBlockingQueue<>(5000),
        r -> new Thread(r, "cache-tx-notify-thread"), (r, e) -> {
        // 把最老的任务拿出来在当前线程执行，把新任务放到队列里
        // 用来保证任务的顺序性
        if (!e.isShutdown()) {
            Runnable oldTask = e.getQueue().poll();
            e.execute(r);
            oldTask.run();
        }
    });
```
