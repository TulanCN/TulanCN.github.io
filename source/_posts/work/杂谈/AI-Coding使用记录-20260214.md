## 前言

2025年已经过去了，如果要简单总结一下2025年到底发生了什么，那我觉得AI coding已经走入了新的阶段——Agentic Coding。

或许Cursor推出Agent模式的时候，没有想到这个Agent编程后面会这么火爆。2025年的后半年，几乎所有人都在用Claude Code或者是ChatGPT的Codex来协助自己的编码开发。

我印象特别深的有两件事情。

第一件事是发生在圣诞节，在国外大家圣诞节毕竟有几天假期，然后在圣诞假期结束之后，几乎我所了解到的很多技术大神，大多都在推特上宣传自己在假期期间vibe coding的一些新项目。

第二件事是今年初的时候，OpenClaw这个项目极其火爆，当然目前他的热度也没下去。作者是全程用Vibe Coding的方式写的这个项目。所以在GitHub上很多人给他提issue，他只能摇摇头说，别问我这些，代码不是我写的。

说这两件事情其实我就是想表述一个核心观点：**目前Vibe Coding出来的项目已经不是玩具项目了**，AI写的代码正在逐渐融入我们日常的生活中。

## 最近用的工具

### Roo Code 与 Claude Code

在2025年8月13号我写了一篇文章，讲解了我当时用的工具。当时我的主力工具是Roo Code，后面我也用它写了不少小玩意儿。

那我是什么时候放弃他的呢，就是刚好在Claude Code推出skills这个概念的时候。

Roo Code 我最初用它的想法是它能够展示上下文长度，在我当时使用的GLM 4.5上，由于我需要控制上下文的用量，超过56K的上下文这个模型就会变得很笨，所以当时我就用Roo Code来做上下文的监控。

后面是有了GLM 4.7，上下文长度已经扩展到200k，和Claude的模型完全一致，这时候Claude Code配合GLM 4.7用起来其实是相当舒服的配合了。

再加上Claude Code上有很好的生态，有大家提供的各种编程工具，我就把我的开发工具完全转向了claude code。

### Speckit

这就是我转向claude code后用的一款工具，它是规范驱动开发的实现，是GitHub开源的一个开发套件。

Speckit其实能在绝大部分coding工具上用，claude code只是它其中一个。

它的核心逻辑就是程序员需要约定各种规范，比如这个项目我们有一些开发约定，你就需要给他提前做好约束，写成一个叫 constitution的文件，所有的规划设计都要遵循constitution。

约定好了规范之后，我们要在规范的框架下进行新功能的设计。设计完还要进行多轮的迭代、反思、评审，确保我们开发的新功能遵循了项目规范。

制定出了一份详细的计划后，speckit会让Claude Code通过subagent的方式并行开发。

这个流程我觉得有点像以前的瀑布开发模型，每个阶段都有详尽的文档，每个阶段完成后才能推进到下一个阶段。

所以这个规范驱动开发它的问题也和瀑布开发模型完全一致，说白了就是很僵硬。

如果是一个需求非常明确的东西，遵循这一套流程开发是非常方便的。

但是很多时候我们在实际开发出第一个原型页面之前并不知道自己想要什么。

这我们绝大部分时候是写了一部分代码，看到效果，然后又想回头去改需求。在这种场景下，规范驱动开发的问题就很大，等于说我们要不停地去修改自己的规范，那么涉及的工程量其实是非常大的。而且多次修改之后，这个规范的质量也就不能保证了

所以才会有原型开发、敏捷之类的其他流程出来。

就我目前的开发方式中，我其实是没有用Speckit的。

### Open Code和Oh-My-OpenCode

在Claude Code火爆的时候，也有一个开源项目出来了，叫做Open Code。如果要对比open code和cloud code的话，Claude Code就像是已经给你准备好了绝大部分东西的windows电脑，而Open Code就像是Linux系统。Open code显然更加开放，成长速度也非常的快，但是它对使用者的要求就很高。

在Open Code极大的自由度之下，就催生出了一个很变态的项目，叫做 Oh-My-OpenCode，简称OMO。

这个OMO项目，核心思路是把不同的任务分配给不同的模型去做。它有一个叫做西西弗斯的调度中枢，会负责把任务解析规划分配成各个子任务，然后每个子任务分派给不同的sub agent去干。

他对每个sub agent定义了一些职责，每个sub agent可以分配特定的模型。比如有一个sub agent是专门用于写前端的，他就让Germini 3.0 pro去做，还有写后端的就让GPT-Codex去做。它的这种方式就要求你有相当多的API账户，

我实际上手试了一下，最后效果不是很好。主要原因还是我没有那么多的国外模型账号，所以我就放弃了这两个工具。

但我觉得这个西西弗斯的agent调度中枢，会成为2026年的爆款。

### compound-engineering-plugin

这是我也用了相当长一段时间的claude code插件。它尝试去解决规范驱动开发中迭代的问题。

它的流程就比较简单，第一个是plan，然后就是work，最后是compound。

关键在最后一步，它会把这一次工作中发现的一些坑点写成solution归档。

然后他还提供了相当多的开发工具，比方说一些skills，subagent配置，还有MCP之类的。

### Superpowers

这是我目前在用的Claude Code插件，这个其实比上面提到的要更强一些，它多了brainstorm阶段。

在一开始的brainstorm阶段，Claude Code会不停地询问你，让你做选择题，这个体验说实话比单纯出一个计划然后进行人工评审要好非常多。
在进行了多轮询问之后，它才会出一个很正式的计划，由于人其实全程参与了这个计划的制定，所以看上去也比较轻松，主要思路很清晰，不会有那种要看长文本的厌恶感。

在计划执行完毕之后可以进一步细化，也就是出一个叫implement plan，制定完实施计划后，才进入实现阶段。

当然，他也提供了相当多的开发工具，属于那种package形式，我只要安装了这一个插件就已经具备了绝大部分的能力。

### OpenClaw

openclaw是我看到了它热度很高，所以尝试安装了一下。实际openclaw是没有参与我的任何工作流程，我用它主要是为了写小说。

这个产品比较有特色的一点是它能给自己搓技能，也就是说，你只要跟他对话，他就可以增加自己的技能。看上去它就像一个会成长的生命体一样。

不知道为什么，就是让他学会各种新技能的时候会有一种莫名其妙的正反馈，也挺容易让人上瘾的。

我是让他写了一整套的小说写作流程。包括那个功法地点的命名工具，小说撰写的pipeline，还有一些审核小说正文内容的工具，这些都是我跟他对话一下就可以直接写出来的。

我前后玩了也有两个礼拜，才把这个小说的写作流程给彻底固化下来。

实际上和他合作的过程确实就像是通过聊天框在和一个员工对话，像是在和他一起规划写一些新的东西。

### GLM-5和kimi-k2.5

这是两个新的国产模型。

GLM-5我关注很久了，出来之后我还升级了我的coding plan，交了将近1900块钱给智谱。

实际GLM-5我用下来的话，能力上确实较GLM-4.7是有进步的，但这个进步的水平我很难评，说实话。就我个人感觉，这一千九百块钱其实是有点花得不太值。

GLM-5最大的进化在于它能够持续执行任务，不会提前中断。所以它就很适合作为AI agent的调度中心模型，他不一定懂一些技术细节，但是他的全局规划能力很棒。

另外一个kimi，这是一个多模态的模型。我目前主要是把它接在了openclaw，他的视觉理解能力和写作能力挺不错的。所以有时候我也会让他改一下我的前端，他的审美说实话在国产模型里面属于第一梯队，我目前还没看到比他更强的。

### Codex

Codex我从开始用到写这篇文章其实也就两天，用下来确实感觉非常惊艳。

我大概可以理解为什么很多人会直接用codeX来做编码开发而放弃claude code。CodeX在大型项目的理解能力上我觉得还是很不错的，尤其是涉及到一些代码重构任务，它的表现要比claude好非常多。

和国产模型就更不用比了，明显比国模要好上一截。

虽然还是有点不甘心，但差距确实存在。对于一些不是那么机密的任务，我还是愿意用一下CodeX的。

### 智谱AI输入法

这个输入法目前用了大概三天。主要是在输入大段大段的提示词的时候，打字速度已经成了限制我工作速度的地方。

它的语音输入能够自动过滤掉我话语里的那些嗯啊之类的语气助词，这样基本上我说出来的话就能直接作为提示词输入到终端里面送给agent了。像今天的这篇文章其实也是通过这个输入法来写的。

不用这种语音输入还真不知道原来我平时说话的时候废话有这么多。

而且我说话的长句比我想的要多得多得多，包括我刚才说这句话的时候发现也是一个长句。

它有个比较好用的功能叫做人设，我可以自定义我语音输入的文字，它要用什么什么样的方式来进行润色。这就给了我很大的自由度，我自定义了一个人设，用来写书面化的博客文字。

（这行之后的文字就是我修改了人设之后出来的，明显表达变得书面化了吧。）

修改前和修改后，句子的核心意思保持不变，但表达方式变得更加书面化，这是一个很好的改进。

## 总结

目前看来，我们的agent coding已经进化到了一个能独立完成大量任务的阶段。

我们现在更关注它能不能把一整个需求给独立完成了，包括前端，后端还有测试。而不是像以前那样我们只是让它写一个短短的函数。

相信2026年这些工具还会继续进化。对比2025年我上一次写文章的时候，我使用的工具基本就固定为Claude Code，但我在Claude Code上尝试了多种编码插件和编码风格。

就目前看来，如果我们希望agent不停的工作，那么我们就必须做到需求、开发、测试的闭环。关键在于最后一步，能让agent自行进行测试和断言，这样就是一个闭环了。这就是为什么TDD是个很有效的Agentic Coding方法。

**一个agent运行的时间越长，消耗的token越多，它能够完成的任务也就越复杂。**

如果一个agent能够在没有任何人介入的情况下运行24小时执行一个任务，哪怕他只能完成一个人正常半天的工作量，这也是非常大的提升，因为这半天的工作量是完全不需要人工参与的。

**将人从工作中解脱出来，这才是最关键的一点。**

目前一个人操作三个agent就会发现很多任务已经没办法规划了，因为用户已经卡在了输入提示词这件事情上。Agent在每一个执行完任务后都会找用户进行确认，三个Agent不停的确认基本上已经占满了一个人所有的工作时间。

试想一下，如果我有10个agent，每个小时这些agent都只回来向用户确认一次，那么是不是一个人就可以同时指挥10个agent了？

未来恐怕一个人要能指挥10到20个agent，那么agent之间互相协作会是一个很重要的方向，agent合作成了一个agent团队。

当我们把一个需求交给一个agent团队时，实际上就是指挥了多个Agent来完成一个超大的任务。这样，基于团队协作的能力，最终能完成的任务应该会是非常厉害的。

从现有的组织架构来看，我作为一个程序员接到的任务其实是团队分配给我的，那么是谁把这个任务分配给我们的团队呢？

跨团队之间往往也有一个更大的任务目标，那这个任务又是从哪里来的呢？

以前的AI coding只是能帮我完成任务，所以实际上是我把团队分配给我的任务分派给了agent。

假设Agent已经能够完美地完成我自己的任务，那么后面让多个Agent协作成为团队，是不是就可以直接把团队的任务完成了呢，也就替代掉了我目前程序员的这个岗位。

团队的任务应该是更上级的公司或者战略拆分下来的，假如有一天Agent团队的能力已经足够强大，多个agent团队可以互相协作，那是不是能够直接把整个战略给全部执行完呢？

恐怕所谓的一人公司，就是这么来的吧。

这就是我之前一直很想写的一篇文章，名为自下而上重构一切。Agent现在正逐步取代人类的能力，每次取代后，人类的参与度会逐渐降低，转而更关注更上层、更抽象的事物。Agent能够不断取代人类的能力，帮助我们处理这些更抽象的事情。那不断取代之后，最后恐怕会出现这种情况：我只要说一句我要看电影，Agent会自动分析我最近的经历，然后分析出我的喜好，当场创作出一部新的电影给我看。

我觉得2025年初我可能会认为这种情况起码要10年以后才会出现，但目前看来，我认为未来五年内可能就会到达这个程度。

而且还有一点，现在我们只是在艺术创作或者说是软件层面走了很远，体力劳动依旧得依靠人力。但具身智能是可以物理上取代人的劳动的，这个恐怕才是能真正将人从繁重的劳动中解脱出来的技术。显然在我们这一代人，大家真的会见到AGI吧。
